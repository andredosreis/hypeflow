## User Story

> Como **agência que recebe leads de formulários web (Tally) além de WhatsApp (Evolution)**, quero **um adapter Tally a normalizar `FORM_RESPONSE` para o mesmo Lead DTO universal já entregue na story 03.1**, para que **(a) leads de qualquer provider apareçam no mesmo CRM com os mesmos campos, (b) o `source.platform = 'form'` permita filtrar e calcular ROI por canal, e (c) ligar um terceiro provider (Meta, Typeform, …) seja mais barato — o pipeline já está provado em duas formas diferentes**.

---

## Contexto

A Story 03.1 entregou o pipeline completo (`auth → rate-limit → adapter → dedup → score-stub → persist → dead-letter`) e o primeiro adapter (Evolution API). Tally é o segundo: payload com forma radicalmente diferente (formulário web — array de `fields` com tipos `INPUT_EMAIL`, `INPUT_PHONE_NUMBER`, `INPUT_TEXT` etc.), assinatura **HMAC-SHA256 base64** (`Tally-Signature` header), não bearer token.

### Por que Tally e não Meta agora

- Meta Lead Ads exige uma chamada à Graph API após o webhook (o webhook só entrega `leadgen_id`); adiciona uma dependência externa síncrona ao pipeline.
- Tally é o caso "puro" — JSON HMAC-signed direto no body. Valida a abstração `Adapter<TPayload>` sem confundir com complexidade de OAuth/Graph.
- Quando Meta vier (story 03.3), o terceiro adapter chega num pipeline já testado em duas formas distintas.

### Estado actual

- Pipeline em `lib/ingestion/*` está pronto e exercitado por Evolution.
- `verifyHmac` em `lib/ingestion/auth.ts` faz hex digest. **Tally usa base64.** Esta story estende o helper com um `encoding?: 'hex' | 'base64'` opcional, default `'hex'` (sem regressão para Evolution — Evolution nem usa HMAC).
- Migração 0008 já cobre todas as colunas necessárias. Esta story **não toca em SQL**.
- `webhook_failures` já existe.

### Não-negociáveis herdados do FDD

Mesmas que 03.1 (FDD §6) — não repetir aqui. Notavelmente:

1. `source.platform = 'form'` em 100 % dos leads persistidos via Tally.
2. `client_id` desconhecido → 200 silencioso.
3. Assinatura HMAC inválida → **401 sem detalhes**, payload **descartado** (não vai para dead-letter — possível ataque).
4. `agency_id` derivado do `client_id` no hub, nunca vem do payload Tally.
5. Score Engine continua stub — `score = 0`, `metadata.score_pending = true`.

## Acceptance Criteria

### Auth (HMAC base64)

- [x] **AC1:** `lib/ingestion/auth.ts::verifyHmac` ganha um quarto argumento opcional `{ encoding?: 'hex' | 'base64' }` (default `'hex'`). Comportamento `'hex'` é **byte-equal** ao actual — Evolution route continua a passar sem alterações.
- [x] **AC2:** Quando `encoding='base64'`:
  - HMAC-SHA256 do `rawBody` é calculado com `secret`, depois codificado em base64.
  - Comparação é constant-time via `timingSafeEqual` sobre os bytes decodificados.
  - Strip opcional do prefixo `sha256=` (Tally **não** usa o prefixo, mas mantemos a lógica consistente para outros providers).

### Adapter

- [x] **AC3:** `lib/ingestion/adapters/tally.ts` exporta `tallyAdapter: Adapter<TallyPayload>` com `provider = 'tally'`.
- [x] **AC4:** `tallyAdapter.verify(req, expectedSecret)` lê `Tally-Signature` (case-insensitive) e delega a `verifyHmac(rawBody, header, secret, { encoding: 'base64' })`. Body raw é passado pelo route handler (já é assim em 03.1, via `AdapterContext.rawText`).
- [x] **AC5:** `tallyAdapter.parse(body, ctx)`:
  - Aceita apenas `eventType === 'FORM_RESPONSE'` — outros tipos retornam `{ skip: true, reason: 'event <type> not handled' }`.
  - Itera `body.data.fields[]` para extrair contacto:
    - **Email**: primeiro field com `type === 'INPUT_EMAIL'` e `value` não-vazio.
    - **Phone**: primeiro field com `type === 'INPUT_PHONE_NUMBER'` e `value` não-vazio.
    - **Name**: primeiro field cujo label (lowercased) inclua `name` ou `nome` ou `nombre`, ou `type === 'INPUT_TEXT'` e for o primeiro `INPUT_TEXT` se nenhum match por label.
  - Se nenhum dos três puder ser extraído → `throw new Error('no contact info')` (route encaminha para dead-letter como `schema_invalid`).
  - **UTM**: extrai de `body.data.fields[]` quaisquer fields cujo `key` ou `label` (lowercased) seja `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`, `utm_term`. Hidden fields da Tally aparecem aqui.
  - `event_id`: usa `body.data.responseId` (Tally garante unicidade global) prefixado com `'tally:'`. Se ausente, fallback para `generateEventId(body, receivedAt, 'tally')`.
  - `source.platform = 'form'`.
  - `metadata`: `{ tally_form_id, tally_form_name, tally_event_id }`.
  - `raw_payload`: o payload Tally completo.
  - `received_at`: `new Date().toISOString()`.

### Route handler

- [x] **AC6:** `app/api/webhooks/tally/[client_id]/route.ts` POST handler segue o mesmo pipeline de 9 passos da Evolution route:
  1. Size guard (1 MB).
  2. Read raw body.
  3. Auth (HMAC) — `tallyAdapter.verify(req, secret)` usando `TALLY_SIGNING_SECRET_<CLIENT_UUID>` do env.
  4. Client lookup → 200 silencioso se não existe.
  5. Rate limit (mesma helper, mesmas thresholds).
  6. JSON parse.
  7. `tallyAdapter.parse` → `{ skip }` ou DTO.
  8. Zod validate DTO (paranoid).
  9. Dedup → score stub → persist; falha de DB → dead-letter + 503.

## Dev Notes

### Como configurar Tally no painel do cliente (não-código, mas referência operacional)

1. No painel Tally do cliente: **Form Settings → Integrations → Webhooks → Add Webhook**.
2. URL: `https://app.hypeflow.com/api/webhooks/tally/<CLIENT_UUID>` (substituir pelo UUID real do client em `clients`).
3. Signing Secret: Tally gera automaticamente. Copia esse valor.
4. No `.env.local` da app (e em Vercel para produção): adicionar `TALLY_SIGNING_SECRET_<CLIENT_UUID_UPPERCASE_UNDERSCORED>=<o valor copiado>`.
   - Ex.: para `client_id = 11111111-1111-1111-1111-111111111111`, a env var é `TALLY_SIGNING_SECRET_11111111_1111_1111_1111_111111111111`.
5. Reiniciar a app (em dev) ou redeploy (em prod). O webhook fica activo de imediato.

### Field-mapping pragmático (Tally → contact)

Tally permite formulários arbitrários. A heurística:

- `INPUT_EMAIL` é semanticamente unívoco — primeiro match vence.
- `INPUT_PHONE_NUMBER` idem.
- `INPUT_TEXT` é ambíguo (pode ser nome, empresa, mensagem…). Heurística: label contém `name | nome | nombre` (case-insensitive), senão o primeiro `INPUT_TEXT` é o "nome" (worst-case, lead vai com nome incorrecto — **não é falha crítica** porque dedup usa email/phone primários).

Quando o cliente quiser controlo fino, story futura adiciona um mapping JSON por client_id.

### Why HMAC-SHA256 base64 (Tally choice)

Tally usa o esquema padrão de webhooks signed: HMAC-SHA256 sobre o raw body, base64 do digest, header `Tally-Signature`. Todos os signing-libraries fazem o mesmo (Stripe usa hex; Slack usa hex; Tally usa base64). O nosso helper agora suporta os dois sem duplicar código.

## Known Gaps (carry-over para stories futuras)

1. **Field mapping configurável por client_id** — heurística actual é "boa o suficiente" para o MVP. Configuração explícita JSON por cliente fica para 03.4+.
2. **Tally hidden fields → UTM** — extracção é por chave/label exacta. Algumas instalações podem usar nomes diferentes (`utm-source` com hífen, `utmSource` camelCase). Esta story só apanha snake_case canónico.
3. **Multi-form por client_id** — mesma chave (Tally signing secret) funciona para todos os forms do mesmo workspace. Se um agência tiver múltiplos workspaces Tally, precisa de chaves distintas. Workaround: criar clients separados, ou story futura adiciona suporte a múltiplos secrets por client.
4. **`is_agency_admin()` policy widening** — herdado de 03.1, ainda pendente.
5. **Real Score Engine** — herdado.
6. **Rate-limit refactor** — herdado.
