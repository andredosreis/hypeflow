## User Story

> Como **manutentor da plataforma**, quero **que a tabela de rate-limit não esteja exposta ao PostgREST de todo**, para que **a tabela seja invisível à API auto-gerada por design e não dependa de policies deny-all para se proteger**.

---

## Contexto

A migration `0005_ai_rate_limits.sql` (story 01.9 / audit C6) criou a tabela `public.ai_rate_limits` como contador sliding-window por IP × rota. A migration deliberadamente deixou RLS off, com a justificação **"Service-role only — não exposto a user code"**. Isto é verdadeiro a nível de aplicação (todos os call sites usam `createServiceClient()`), mas não a nível de Postgres: qualquer tabela em `public.*` é auto-exposta pelo PostgREST do Supabase, e portanto a `anon`/`authenticated` keys (que vão no bundle frontend) podem atingir a tabela directamente.

O linter do Supabase emite ERROR `rls_disabled_in_public` por esta razão.

### Fix imediato (já aplicado fora desta story)

A migration `0009_ai_rate_limits_rls.sql` (`hypeflow-os/supabase/migrations/0009_ai_rate_limits_rls.sql`) endurece `public.ai_rate_limits`:

- `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`
- Duas policies deny-all explícitas (`TO anon`, `TO authenticated`) — `USING (false) WITH CHECK (false)`
- `REVOKE ALL ON TABLE ... FROM anon, authenticated`

O service-role bypassa RLS por design no Supabase (atributo `BYPASSRLS`), portanto `apps/hypeflow/lib/api/rate-limit.ts` continua a funcionar inalterado. O linter pára de emitir ERROR.

### Por que esta story existe

A docs do Supabase recomenda explicitamente, para tabelas internas / system-only, **mover a tabela para um schema fora de `public`** (ex.: `private`) em vez de depender de RLS deny-all:

> "Custom schemas can be created to organize tables, such as establishing a private schema to hide specific tables from your API."  
> — supabase.com/docs/guides/database/tables (consultado via Context7, 2026-05-06)

Schemas que não estão em `db.schemas` da config do PostgREST **não são expostos pela auto-gerada API**, ponto. Isto remove a tabela inteiramente da superfície de ataque pela anon key, sem precisar de policies para a defender.

Trade-off: requer ajustes mínimos em código (`supabase.schema('private').from('ai_rate_limits')`) e uma migration cuidada (mover a tabela, recriar índice, manter dados live).

### Decisão de scope

| Decisão | Escolha |
|---|---|
| Estratégia de migração | Criar schema `private` se não existir; `ALTER TABLE public.ai_rate_limits SET SCHEMA private`; remover policies/grants de `public` (já não necessárias depois do move) |
| Preservação de dados | A tabela só guarda timestamps de rate-limit dos últimos 60s — não há valor histórico. Migration pode ser feita com tabela vazia ou não — ambas são seguras |
| Schema name | `private` (alinhado com docs canónicos do Supabase) |
| Outras tabelas neste move | Apenas `ai_rate_limits` nesta story. Se aparecerem outras candidatas (ex.: tabelas de cron/cleanup futuras), abrir story própria |

## Technical Notes

### Migração detalhada (template)

```sql
-- supabase/migrations/NNNN_ai_rate_limits_to_private_schema.sql
-- Story 01.14 — move public.ai_rate_limits to private schema (follow-up to 01.9 / 0009).
--
-- Why: 0009 fixed the linter ERROR by enabling RLS deny-all on the table. The cleaner
-- pattern (per Supabase docs) for purely internal/system tables is to move them out of
-- `public` entirely so PostgREST never exposes them. This story does that.
--
-- Safety: ai_rate_limits only stores per-IP request timestamps for the last ~60s.
-- Even if rows are lost during the schema move, the worst outcome is that the next
-- 20 requests per (IP, route) are not counted. No data of value is at risk.

begin;

-- 1. Ensure schema exists.
create schema if not exists private;

-- 2. Clean up the deny-all policies from migration 0009 — they target a `public` table
--    that's about to move. Keeping them attached on the original schema is harmless,
--    but explicit is better.
drop policy if exists "ai_rate_limits_deny_anon"          on public.ai_rate_limits;
drop policy if exists "ai_rate_limits_deny_authenticated" on public.ai_rate_limits;

-- 3. Move the table. Index, PK, and column comments follow the table automatically.
alter table public.ai_rate_limits set schema private;

-- 4. Defensive revoke — anon/authenticated never had grants in `private` anyway,
--    but state it explicitly so future readers don't have to derive it.
revoke all on table private.ai_rate_limits from anon, authenticated;

-- 5. Update table comment.
comment on table private.ai_rate_limits is
  'Sliding-window counter for /api/ai/* rate limits. Story 01.9 / 01.14. '
  'Lives in `private` schema — not exposed to PostgREST. Service-role only.';

commit;
```

### Refactor exemplo em `rate-limit.ts`

```ts
// Antes
const { count, error: countErr } = await supabase
  .from('ai_rate_limits')
  .select('id', { count: 'exact', head: true })
  .eq('bucket_key', bucketKey)
  .gte('ts', windowStart.toISOString())

// Depois
const { count, error: countErr } = await supabase
  .schema('private')
  .from('ai_rate_limits')
  .select('id', { count: 'exact', head: true })
  .eq('bucket_key', bucketKey)
  .gte('ts', windowStart.toISOString())
```

Mesma transformação nos outros 2 call sites (oldest lookup e insert).

### Verificar `Exposed schemas` no Supabase Dashboard

`Project → Settings → API → Exposed schemas` (default: `public, graphql_public`). Se `private` for adicionado aqui, o move perde valor — a tabela voltaria a ser exposta. Esta verificação deve ficar registada nas QA notes.

### Por que `private` e não outro nome

Apenas convenção. Os docs do Supabase usam `private` como exemplo canónico. Outras opções (`internal`, `infra`) seriam funcionalmente equivalentes. Manter `private` para alinhamento com docs e para que outros maintainers reconheçam o padrão imediatamente.
