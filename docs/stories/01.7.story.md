## User Story

> Como **sistema HYPE Flow OS**, quero **ler identidade e autorização do utilizador (role, agency_id, client_id) a partir do JWT** em vez de ir à base de dados a cada request, para que **o middleware edge e o contexto tRPC não dependam de roundtrips DB e a plataforma aguente carga alta sem degradar latência**.

---

## Contexto

A auditoria [context7-audit-report.md](../architecture/context7-audit-report.md) (v2, 2026-04-21) identificou dois problemas críticos relacionados:

- **CR-01:** `apps/hypeflow/lib/supabase/middleware.ts:42-49` faz query `SELECT FROM users` a cada navegação em rotas protegidas (edge → DB roundtrip).
- **CR-03:** `apps/hypeflow/server/trpc.ts:32-66` faz 2-3 queries redundantes (`users` + `client_users`) por request tRPC só para resolver identidade.

Ambos desaparecem se `role`, `agency_id` e `client_id` viverem nos claims do access token Supabase.

A solução oficial é o [Supabase Auth Hook `custom_access_token_hook`](https://supabase.com/docs/guides/auth/auth-hooks) — uma Postgres function que o Supabase Auth invoca antes de emitir o JWT e que permite injectar claims customizados.

## Acceptance Criteria

### Hook & claims

- [ ] **AC1:** Criar migração Postgres que define função `public.custom_access_token_hook(event jsonb) returns jsonb` injectando no token:
  - `user_type`: `"agency"` | `"client"` | `null`
  - `agency_id`: UUID ou `null`
  - `client_id`: UUID ou `null` (apenas para `user_type = 'client'`)
  - `role`: string (`agency_admin` | `agency_manager` | `agency_viewer` | `client_admin` | `client_viewer`) ou `null`
- [ ] **AC2:** Activar o hook no `auth.hooks` do projecto Supabase via migração ou painel (documentar o procedimento em `docs/architecture/`).
- [ ] **AC3:** Claims ficam disponíveis em `user.app_metadata` (não `user_metadata` — este é mutável pelo cliente).
- [ ] **AC4:** Função é `SECURITY DEFINER` e apenas a role `supabase_auth_admin` pode invocá-la (grant explícito).

### Middleware edge

- [ ] **AC5:** `apps/hypeflow/lib/supabase/middleware.ts` deixa de invocar `supabase.from('users').select(...)`.
- [ ] **AC6:** Resolução de `isAgencyUser` passa a ler directo do token:
  ```ts
  const { data: { user } } = await supabase.auth.getUser()
  const isAgencyUser = user?.app_metadata?.user_type === 'agency'
  ```
- [ ] **AC7:** Todas as redirecções (`/admin` ↔ `/client` ↔ `/login`) continuam a funcionar como antes.
- [ ] **AC8:** Métrica edge: latência p95 do middleware cai de baseline actual para ≤ 50ms (medir antes e depois).

### Contexto tRPC

- [ ] **AC9:** `apps/hypeflow/server/trpc.ts::createTRPCContext` deixa de invocar `supabase.from('users')` e `supabase.from('client_users')`.
- [ ] **AC10:** `agencyUser` e `clientUser` são derivados dos claims sem DB query adicional (apenas `auth.getUser()` continua).
- [ ] **AC11:** Middlewares `enforceAgencyUser` e `enforceClientUser` continuam a funcionar sem mudança de contrato.

### Invalidação e consistência

- [ ] **AC12:** Ao mudar `role`, `is_active` ou desactivar user, o token corrente fica inválido dentro de ≤ 60s. Testar:
  - Opção A: TTL curto no access token (~60s) com refresh token padrão
  - Opção B: Invocar `auth.admin.signOut(user_id)` quando role muda
- [ ] **AC13:** Se `is_active = false`, queries às rotas protegidas falham com 401 (via RLS ou enforcement no middleware).

### Retrocompatibilidade

- [ ] **AC14:** Utilizadores existentes em produção recebem os novos claims ao próximo refresh do token (sem re-login manual). Documentar janela de transição.

## Technical Notes

### Exemplo da função Postgres

```sql
create or replace function public.custom_access_token_hook(event jsonb)
returns jsonb
language plpgsql
stable
as $$
declare
  claims jsonb;
  agency_row record;
  client_row record;
begin
  claims := event->'claims';

  -- Tentar agency user
  select id, agency_id, role into agency_row
  from public.users
  where id = (event->>'user_id')::uuid and is_active = true;

  if found then
    claims := jsonb_set(claims, '{app_metadata}', coalesce(claims->'app_metadata', '{}'::jsonb));
    claims := jsonb_set(claims, '{app_metadata,user_type}', '"agency"');
    claims := jsonb_set(claims, '{app_metadata,agency_id}', to_jsonb(agency_row.agency_id));
    claims := jsonb_set(claims, '{app_metadata,role}',      to_jsonb(agency_row.role));
    return jsonb_set(event, '{claims}', claims);
  end if;

  -- Tentar client user
  select client_id, agency_id into client_row
  from public.client_users
  where id = (event->>'user_id')::uuid and is_active = true;

  if found then
    claims := jsonb_set(claims, '{app_metadata}', coalesce(claims->'app_metadata', '{}'::jsonb));
    claims := jsonb_set(claims, '{app_metadata,user_type}', '"client"');
    claims := jsonb_set(claims, '{app_metadata,agency_id}', to_jsonb(client_row.agency_id));
    claims := jsonb_set(claims, '{app_metadata,client_id}', to_jsonb(client_row.client_id));
    return jsonb_set(event, '{claims}', claims);
  end if;

  return event;
end;
$$;

grant execute on function public.custom_access_token_hook to supabase_auth_admin;
revoke execute on function public.custom_access_token_hook from authenticated, anon, public;
```

### Exemplo do middleware refactorizado

```ts
const { data: { user } } = await supabase.auth.getUser()
const userType = user?.app_metadata?.user_type as 'agency' | 'client' | undefined

if (!user && (pathname.startsWith('/admin') || pathname.startsWith('/client'))) {
  return NextResponse.redirect(new URL('/login', request.url))
}

if (userType === 'agency' && pathname.startsWith('/client')) {
  return NextResponse.redirect(new URL('/admin/dashboard', request.url))
}

if (userType === 'client' && pathname.startsWith('/admin')) {
  return NextResponse.redirect(new URL('/client/dashboard', request.url))
}
```

### Rollback

Se o hook falhar em produção: desactivar o hook no painel Supabase → tokens voltam a não ter claims customizados. O código refactorizado deve tratar claims em falta como "utilizador não autenticado para rota protegida" (fail-closed).
