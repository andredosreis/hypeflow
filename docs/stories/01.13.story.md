**Epic:** EPIC-01 — Autenticação & Multi-tenant Setup

**Predecessor:** [Story 01.12 — C2 service role / workspace bootstrap](01.12.story.md)

## User Story

> Como **operador de uma agência cujos URLs de portal podem ser partilhados com clientes (e potencialmente reencaminhados)**, quero **que cada token de portal seja um valor aleatório de alta entropia, com TTL e revogação geridos no servidor, e que o acesso ao portal só carregue dados depois de o token ser validado server-side**, para que **um cliente que perde acesso (rescisão, partilha indevida) seja revogado em segundos sem expor uma forma de o atacante regenerar o seu token a partir do `clientId`**.

---

## Contexto

O [Test Audit Report 2026-04-24](../audits/test-audit-report-2026-04-24.md), achado **C5**, identificou:

1. **Token deterministico via hash polinomial** (`apps/hypeflow/app/(admin)/admin/clientes/page.tsx:376-383`) — qualquer pessoa que conheça o `clientId` recompõe o token executando o mesmo hash. Os `clientIds` vazam por múltiplos canais.
2. **Token contém o `clientId` em plaintext** — apenas trim front 8 + back 16 char e o atacante extrai o ID.
3. **Sem validação server-side** — `app/portal/[token]/page.tsx` é `'use client'`. Carrega `getMockPortalData(token)` baseado em `token.charCodeAt(0) % 6` — sem lookup à DB, sem verificação de expiração, sem auth check.
4. **Footer afirma "Acesso protegido por token · SSL"** — mentira em produção.

Hoje a página apenas mostra mock data, mas o esquema do token é inseguro e tornar-se-á CRITICAL no instante em que um endpoint real `/api/portal/[token]` for ligado a dados reais de cliente.

### Decisão de scope (alinhada com user)

| Decisão | Escolha |
|---|---|
| Backend de token | **Scheme A do audit — DB-stored opaque token, hash SHA-256 guardado, raw mostrado uma vez** |
| TTL default | **30 dias** (renovável por geração de novo token; revoga a previous) |
| Tokens activos por cliente | **Um (1) único activo** — gerar novo automaticamente revoga o anterior |
| Geração | tRPC procedure `admin.portalTokens.generate` (agency-auth, scope a `agency_id` do utilizador) |
| Revogação | tRPC procedure `admin.portalTokens.revoke` |
| Página `/portal/[token]` | **Convertida para Server Component wrapper** que valida server-side; o cliente UI atual fica como Client Component filho |
| Phase | Atómico (não phase A/B) — esta story remove `derivePortalToken` e substitui pelo novo fluxo numa só PR. As mock data permanecem (sem dados reais de cliente nesta story) — apenas o esquema de token é endurecido |

## Technical Notes

### Estrutura alvo de `lib/portal/tokens.ts`

```ts
import { randomBytes, createHash } from 'crypto'
import { createServiceClient } from '@/lib/supabase/server'

export function generateRawToken(): string {
  return randomBytes(32).toString('base64url')
}

export function hashToken(raw: string): string {
  return createHash('sha256').update(raw).digest('hex')
}

export interface PortalTokenContext {
  client_id: string
  agency_id: string
  token_id: string
}

export async function validatePortalToken(rawToken: string): Promise<PortalTokenContext | null> {
  if (!rawToken || rawToken.length < 20) return null

  const tokenHash = hashToken(rawToken)
  const service = await createServiceClient()

  const { data } = await service
    .from('portal_tokens')
    .select('id, client_id, agency_id, expires_at, revoked_at')
    .eq('token_hash', tokenHash)
    .maybeSingle()

  if (!data) return null
  if (data.revoked_at) return null
  if (new Date(data.expires_at) <= new Date()) return null

  // Best-effort last_used_at update, do not block response
  void service
    .from('portal_tokens')
    .update({ last_used_at: new Date().toISOString() })
    .eq('id', data.id)
    .then(undefined, () => {})

  return { token_id: data.id, client_id: data.client_id, agency_id: data.agency_id }
}
```

-- DB-stored opaque tokens. Token raw shown to agency user once at generation;
-- hash is what's persisted. Lookup is via service role.

begin;

create table public.portal_tokens (
  id           uuid primary key default gen_random_uuid(),
  client_id    uuid not null references public.clients(id) on delete cascade,
  agency_id    uuid not null references public.agencies(id) on delete cascade,
  token_hash   text not null unique,
  created_by   uuid not null references public.users(id) on delete restrict,
  created_at   timestamptz not null default now(),

  last_used_at timestamptz,
  revoked_at   timestamptz,
  label        text
);

create index portal_tokens_active_hash on public.portal_tokens (token_hash) where revoked_at is null;
create index portal_tokens_client_id on public.portal_tokens (client_id);

alter table public.portal_tokens enable row level security;

create policy "agency_portal_tokens_read" on public.portal_tokens
  for select to authenticated
  using (agency_id = get_user_agency_id());

  for all to authenticated
  using (agency_id = get_user_agency_id() and is_agency_admin())

### tRPC procedures (esqueleto)

```ts
// apps/hypeflow/server/routers/admin/portal-tokens.ts
import { z } from 'zod'
import { TRPCError } from '@trpc/server'
import { agencyProcedure, router } from '@/server/trpc'
import { generateRawToken, hashToken } from '@/lib/portal/tokens'

const DEFAULT_TTL_DAYS = 30

export const portalTokensRouter = router({
  generate: agencyProcedure
    .input(z.object({
      clientId: z.string().uuid(),
      label: z.string().max(60).optional(),
      ttlDays: z.number().int().min(1).max(365).optional(),
    }))
    .mutation(async ({ ctx, input }) => {
      // Confirm the client belongs to caller's agency (RLS would block anyway)
      const { data: client } = await ctx.supabase
        .from('clients')
        .select('id, agency_id')
        .eq('id', input.clientId)
        .maybeSingle()
      if (!client || client.agency_id !== ctx.user.agencyId) {
        throw new TRPCError({ code: 'FORBIDDEN', message: 'Client not in your agency' })
      }

      const raw = generateRawToken()
      const tokenHash = hashToken(raw)
      const ttlDays = input.ttlDays ?? DEFAULT_TTL_DAYS
      const expiresAt = new Date(Date.now() + ttlDays * 86_400_000).toISOString()

      // Service role for the write — RLS write policy requires is_agency_admin
      // which complicates owner-role users; we use service role and verify
      // agency match above.
      const service = await createServiceClient()

      // Revoke previous active tokens for this client
      await service.from('portal_tokens')
        .update({ revoked_at: new Date().toISOString() })
        .eq('client_id', input.clientId)
        .is('revoked_at', null)

      const { data, error } = await service.from('portal_tokens').insert({
        client_id: input.clientId,
        agency_id: client.agency_id,
        token_hash: tokenHash,
        created_by: ctx.user.id,
        expires_at: expiresAt,
        label: input.label ?? null,
      }).select('id, expires_at').single()

      if (error || !data) throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR', message: error?.message ?? 'Insert failed' })

      return { id: data.id, rawToken: raw, expiresAt: data.expires_at }
    }),

  revoke: agencyProcedure
    .input(z.object({ tokenId: z.string().uuid() }))
    .mutation(async ({ ctx, input }) => {
      const { error } = await ctx.supabase
        .from('portal_tokens')
        .update({ revoked_at: new Date().toISOString() })
        .eq('id', input.tokenId)
      if (error) throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR', message: error.message })
      return { ok: true }
    }),

  list: agencyProcedure
    .input(z.object({ clientId: z.string().uuid() }))
    .query(async ({ ctx, input }) => {
      const { data, error } = await ctx.supabase
        .from('portal_tokens')
        .select('id, label, created_at, expires_at, last_used_at, revoked_at')
        .eq('client_id', input.clientId)
        .order('created_at', { ascending: false })
      if (error) throw new TRPCError({ code: 'INTERNAL_SERVER_ERROR', message: error.message })
      return data ?? []
    }),
})
```

> Skeleton — actual `agencyProcedure` and `ctx.user.agencyId` shape may need adapting to the existing tRPC setup (`apps/hypeflow/server/trpc.ts`). Tasks include reading and matching that.

### Por que service role no insert/revoke

A policy `agency_portal_tokens_write` exige `is_agency_admin()`. Owners (role `'owner'`) deveriam funcionar via `is_agency_admin`, mas a função actual em `0002_rls_policies.sql:20` retorna `role = 'agency_admin'` strict. Para evitar widening de RLS nesta story (e potencialmente afectar outras tabelas), usamos service role no insert/revoke depois de validar `agency_id` em código. List via anon-role + RLS está OK porque é SELECT (read policy é mais permissiva).

Alternative considerada: ajustar `is_agency_admin` para `role IN ('owner', 'agency_admin')`. Decisão: fora de scope desta story — abrir story RLS-cleanup separada.
