## User Story

> Como **agência em produção com utilizadores reais**, quero **que o middleware de autenticação não devolva 500 em erros transitórios do Supabase, e que o "preview mode" só possa ser activado por uma flag explícita (não por uma substring na URL)**, para que **eu não perca disponibilidade em incidentes Supabase nem perca a auth layer inteira por um typo num env var em Vercel**.

---

## Contexto

O [Test Audit Report 2026-04-24](../audits/test-audit-report-2026-04-24.md), achado **C1**, identificou três problemas em três ficheiros de middleware:

1. **Sem `try/catch`** à volta de `supabase.auth.getUser()` e da query `from('users').select(...).single()`. Qualquer erro transitório do Supabase → Next.js 500 para todos os requests que correspondem ao matcher (efectivamente todo o site). É **risco de disponibilidade**, não de bypass.
2. **Preview gate fail-open** em `apps/hypeflow/lib/supabase/middleware.ts:27-28` e `apps/agency/lib/supabase/middleware.ts:28`:
   ```ts
   const isPreview = process.env.NEXT_PUBLIC_SUPABASE_URL?.includes('placeholder')
   if (isPreview) return supabaseResponse
   ```
   Se `NEXT_PUBLIC_SUPABASE_URL` alguma vez contiver a substring `"placeholder"` numa deployment de produção (Vercel env mal configurado, `.env.local` que vaza para o build de prod), **toda a auth layer é ignorada** — utilizadores anónimos chegam a `/admin/*` e `/client/*`. É bypass por configuração, classe de bug "fica fino no preview e falha aberto em prod".
3. **Routing quirk de utilizador agência inactivo** em `apps/hypeflow/lib/supabase/middleware.ts:42-63`: um utilizador cujo `users` row tem `is_active=false` falha o `.eq('is_active', true).single()`, é tratado como client user (linha 49) e redireccionado para `/client/dashboard`. O path `/client/*` vive noutro app (`apps/portal/`); o redirect aterra no nada — **mas** se `/client/dashboard` alguma vez for adicionado ao app `hypeflow`, um utilizador agência soft-deleted será desviado para a UI de portal sobre os seus próprios dados.

**Ficheiros afectados:**
- `apps/hypeflow/lib/supabase/middleware.ts` (deployed unified app — primary fix surface)
- `apps/agency/lib/supabase/middleware.ts` (legacy/secondary — paridade de código para evitar drift)
- `apps/portal/middleware.ts` (deployed client portal — sem preview gate, mas também sem `try/catch`)

**Não-exploit hoje:** o `try/catch` é availability, e o preview-gate exige que alguém configure mal o env. Mas é HIGH porque o blast radius é "site inteiro 500" ou "auth desligada", e ambos são preveníveis com pequenas mudanças.

### Trade-off — testes nesta story

Conforme regra `@dev customization` ("tests in same PR"), esta story inclui testes Vitest para `apps/hypeflow/lib/supabase/middleware.ts` (a única app com vitest configurado pela story 01.10). `apps/agency` e `apps/portal` recebem o mesmo padrão de fix mas sem testes próprios — adicionar Vitest aos outros dois apps é trabalho de outra story (01.10 fê-lo apenas para `hypeflow`).

## Technical Notes

### Padrão alvo — `apps/hypeflow/lib/supabase/middleware.ts`

```ts
import { createServerClient } from '@supabase/ssr'
import { NextResponse, type NextRequest } from 'next/server'

const PROTECTED_PATH_PREFIXES = ['/admin', '/client'] as const

function isProtectedPath(pathname: string) {
  return PROTECTED_PATH_PREFIXES.some(p => pathname.startsWith(p))
}

export async function updateSession(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return request.cookies.getAll() },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))
          supabaseResponse = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          )
        },
      },
    }
  )

  // Explicit preview gate — fail-closed unless flag is set
  if (process.env.NEXT_PUBLIC_PREVIEW_MODE === 'true') {
    return supabaseResponse
  }

  const { pathname } = request.nextUrl

  try {
    const { data: { user } } = await supabase.auth.getUser()

    if (!user && isProtectedPath(pathname)) {
      const url = request.nextUrl.clone()
      url.pathname = '/login'
      return NextResponse.redirect(url)
    }

    if (user) {
      const { data: agencyRow } = await supabase
        .from('users')
        .select('id, is_active')
        .eq('id', user.id)
        .maybeSingle()

      const isAgencyUser = !!agencyRow
      const isAgencyUserActive = agencyRow?.is_active === true

      // Soft-deleted agency user — keep them out of both areas
      if (isAgencyUser && !isAgencyUserActive) {
        if (pathname === '/login') return supabaseResponse
        const url = request.nextUrl.clone()
        url.pathname = '/login'
        url.searchParams.set('status', 'account-disabled')
        return NextResponse.redirect(url)
      }

      if (isAgencyUserActive && pathname.startsWith('/client')) {
        const url = request.nextUrl.clone()
        url.pathname = '/admin/dashboard'
        return NextResponse.redirect(url)
      }

      if (!isAgencyUser && pathname.startsWith('/admin')) {
        const url = request.nextUrl.clone()
        url.pathname = '/client/dashboard'
        return NextResponse.redirect(url)
      }

      if (pathname === '/login') {
        const url = request.nextUrl.clone()
        url.pathname = isAgencyUserActive ? '/admin/dashboard' : '/client/dashboard'
        return NextResponse.redirect(url)
      }
    }

    return supabaseResponse
  } catch (err) {
    console.error('[middleware] session error', { err, path: pathname })
    if (isProtectedPath(pathname)) {
      const url = request.nextUrl.clone()
      url.pathname = '/login'
      url.searchParams.set('error', 'session')
      return NextResponse.redirect(url)
    }
    return supabaseResponse
  }
}
```

### Por que `maybeSingle` em vez de `single()` com `.eq('is_active', true)`

`single()` lança quando 0 rows; também lança quando >1 rows. O comportamento actual (`.eq('is_active', true).single()`) não distingue "não existe" de "existe-mas-inactivo" — ambos caem no caminho `data: null` e são tratados como client user. Trocar para `.select('id, is_active').eq('id', user.id).maybeSingle()` retorna a row se existir (qualquer `is_active`) ou `null` se não existir — permite o branching explícito do AC8.

### `agency` middleware — não tem lookup de users

`apps/agency/lib/supabase/middleware.ts` apenas faz `getUser()` + path-based redirect. O `try/catch` envolve só esse bloco. Não precisa do branching de inactive-user (não chega a olhar para `users`).

### `portal` middleware — todas as rotas protegidas

`apps/portal/middleware.ts` faz `getUser()` + redirect a `/login` se ausente. O `try/catch` envolve esse bloco; em catch, redirect a `/login?error=session` se path **não** começa por `/login` ou `/auth`.
