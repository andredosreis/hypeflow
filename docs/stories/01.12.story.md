## User Story

> Como **agência em produção**, quero **que o bootstrap de workspace use o cliente Supabase autenticado (anon + RLS) em vez de service role, e que só corra na primeira navegação por utilizador (não em todas)**, para que **(a) cada navegação não disparar 5+ queries service-role contra a DB e (b) não exista nenhum caminho anónimo que escreva via service role**.

---

## Contexto

O [Test Audit Report 2026-04-24](../audits/test-audit-report-2026-04-24.md), achado **C2**, identificou três problemas em `apps/hypeflow/lib/bootstrap/workspace.ts`:

1. **Ramo anónimo escreve via service role** (`workspace.ts:41-228`). Após C1 (story 01.11), o middleware bloqueia anónimos em `/admin/*`, mas a função ainda contém o caminho `if (!user)` que cria/popula uma "demo-workspace" partilhada via service role. Código vivo, blast radius elevado se C1 alguma vez regredir.
2. **Ramo autenticado também usa service role** (`workspace.ts:230-433`). Cada navegação a uma página `/admin/*` dispara 5+ queries service-role (users, clients, pipeline_stages, leads, traffic_metrics) e potencialmente INSERTs. RLS é ignorado. Aumenta superfície de ataque se a função alguma vez aceitar input do utilizador.
3. **`hasSupabase` silent fallback** (`workspace.ts:23-34`, `app/(admin)/admin/trafego/page.tsx:6-13`). Se uma env var contiver `'placeholder'` ou estiver ausente, a página serve dados demo silenciosamente em vez de falhar — exactamente o tipo de falha que "parece bem em preview e quebra em prod".

**Combinação com C1:** após o fix da story 01.11, os anónimos não atingem a função (middleware redireciona). Mas o ramo dead code permanece. A combinação C1+C2 só fica resolvida quando este ramo é removido e o ramo autenticado deixa de usar service role para tudo.

### Decisões de scope (a confirmar com user)

| Decisão | Escolha proposta | Razão |
|---|---|---|
| Ramo anónimo da bootstrap | **Eliminar inteiramente** | Após C1, é dead code. Mover para `npm run seed:demo` é trabalho separado (story futura) — se for preciso voltar a popular dados demo, faz-se intencionalmente. |
| Demo seed em produção | **Remover do bootstrap autenticado** (clients/leads/traffic_metrics) | A bootstrap deve criar **só agência + linha em users** na primeira navegação. Popular pipeline_stages é obrigatório para funcionalidade básica; o resto (leads de exemplo, 30 dias de traffic_metrics) é demo, não core, e não deve correr na bootstrap. |
| Service role isolado | **Apenas para criar a `agency` row + `users` upsert na primeira navegação** | Todas as outras operações usam o cliente anon-role com RLS. Comentário no service-role helper a explicar porquê. |
| `bootstrapped` flag | **Adicionar coluna `users.bootstrapped boolean not null default false`** | Layout `(admin)/layout.tsx` faz `select bootstrapped`; se true, skip; se false, corre o bootstrap e marca true. Reduz queries por navegação de 5+ para 1. |
| Demo mode | **Substituir `URL.includes('placeholder')` por `NEXT_PUBLIC_DEMO_MODE === 'true'`** explícito | Mesmo padrão da preview gate (C1). Default fail-closed. |
| Env validation | **Throw em produção quando `SUPABASE_SERVICE_ROLE_KEY` falta**; warn em dev | Eliminar o silent fallback `agencyId: 'demo-agency-id'`. |

  1. `INSERT INTO agencies` (primeira criação) — RLS não permite anon-role criar uma agência onde o `agency_id` ainda não existe nas suas claims.
  2. `UPSERT INTO users` com `agency_id` e `bootstrapped=true` — para inicializar a row do utilizador.
- [x] **AC8:** Cada chamada a `createServiceClient()` está envolvida num helper `bootstrapNewWorkspace(user)` com comentário explícito: *"Service role required: anon-role cannot create the first row of an agency the user is not yet a member of."*

- [x] **AC10:** A população de `pipeline_stages` por defeito (5 stages) ocorre **dentro** do helper `bootstrapNewWorkspace` (uma única vez por utilizador), também via service role pelo mesmo motivo (criação inicial). ## Technical Notes

### Estrutura alvo de `workspace.ts`

```ts
import { createClient, createServiceClient } from '@/lib/supabase/server'

const DEMO_RESPONSE = { user: null, agencyId: 'demo-agency-id', clientId: 'preview-client-1' }
const DEFAULT_STAGES = [/* same 5 stages */]

function isDemoMode() {
  if (process.env.NEXT_PUBLIC_DEMO_MODE !== 'true') return false
  if (process.env.NODE_ENV === 'production') {
    throw new Error('NEXT_PUBLIC_DEMO_MODE=true is not allowed in production')
  }
  return true
}

export async function ensureWorkspaceForCurrentUser() {
  if (isDemoMode()) return DEMO_RESPONSE

  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  if (!user) {
    throw new Error('Bootstrap requires authenticated user — middleware should have redirected')
  }

  // Anon-role read with RLS — fast path
  const { data: profile } = await supabase
    .from('users')
    .select('agency_id, bootstrapped')
    .eq('id', user.id)
    .maybeSingle()

  if (profile?.bootstrapped && profile.agency_id) {
    // Already initialized; resolve clientId via anon-role + RLS
    const { data: client } = await supabase
      .from('clients')
      .select('id')
      .eq('agency_id', profile.agency_id)
      .order('created_at', { ascending: true })
      .limit(1)
      .maybeSingle()

    return { user, agencyId: profile.agency_id, clientId: client?.id ?? null }
  }

  // First-time path or unfinished bootstrap
  return bootstrapNewWorkspace(user, profile)
}

/**
 * Service role required: anon-role cannot:
 *  - create the first agency row (RLS would block — user has no agency_id yet)
 *  - upsert their own users row with the new agency_id link
 *  - insert default pipeline_stages tied to the new agency
 *
 * All other operations (reading clients, leads, traffic_metrics) use the
 * authenticated anon-role client and respect RLS.
 */
async function bootstrapNewWorkspace(
  user: User,
  existingProfile: { agency_id: string | null; bootstrapped: boolean } | null,
) {
  const service = await createServiceClient()

  let agencyId = existingProfile?.agency_id ?? null
  if (!agencyId) {
    agencyId = crypto.randomUUID()
    const seed = user.email?.split('@')[0] ?? 'agency'
    const slug = `${slugify(seed)}-${agencyId.slice(0, 6)}`

    await service.from('agencies').insert({
      id: agencyId, name: `Agencia ${seed}`, slug, plan: 'starter', settings: {},
    })
  }

  await service.from('users').upsert({
    id: user.id,
    agency_id: agencyId,
    full_name: (user.user_metadata?.full_name as string | undefined) ?? user.email?.split('@')[0],
    email: user.email,
    role: 'owner',
    is_active: true,
    bootstrapped: true,
    last_login: new Date().toISOString(),
  })

  // pipeline_stages default — only if missing
  const { data: existingStages } = await service
    .from('pipeline_stages')
    .select('id').eq('agency_id', agencyId).limit(1)

  if (!existingStages?.length) {
    await service.from('pipeline_stages').insert(
      DEFAULT_STAGES.map(s => ({ ...s, id: crypto.randomUUID(), agency_id: agencyId }))
    )
  }

  return { user, agencyId, clientId: null }
}
```

### Migração proposta

```sql
-- supabase/migrations/NNNN_users_bootstrapped.sql
-- Story 01.12 / audit C2: short-circuits the bootstrap function on every page load.
-- See apps/hypeflow/lib/bootstrap/workspace.ts.

begin;

alter table public.users
  add column if not exists bootstrapped boolean not null default false;

comment on column public.users.bootstrapped is
  'Set to true after ensureWorkspaceForCurrentUser has finished initial agency + pipeline_stages setup. Used by app/(admin)/layout.tsx to short-circuit subsequent page loads.';

-- Backfill: any existing user row with an agency_id is already bootstrapped.
update public.users set bootstrapped = true where agency_id is not null;

commit;
```

### RLS considerations

A leitura inicial em `(admin)/layout.tsx` usa anon-role + RLS. A política RLS do `users` table tem que permitir SELECT da própria row (auth.uid() = id). Confirmar em `multi-tenancy.md` ou ler a política existente. Se a política não permitir, esta story precisa de adicionar/ajustar a policy — registar como sub-task.

### Cuidado com a regra "backfill antes de constraint"

A migração adiciona `NOT NULL DEFAULT false`. Para tabela com produção, isto é seguro (default cobre rows existentes). O backfill subsequente só corrige semanticamente quem deveria ser `true`. Não há novo `CHECK` ou `FOREIGN KEY` a adicionar — só uma coluna.
