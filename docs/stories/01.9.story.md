O [Test Audit Report 2026-04-24](../audits/test-audit-report-2026-04-24.md), achado **C6** (adicionado na Nota de Verificação Pós-Auditoria), identificou que os três endpoints `/api/ai/*` no app unificado são **públicos e sem autenticação**, usando a `ANTHROPIC_API_KEY` do servidor em representação de chamadores anónimos:

**Exploit activo:** um loop `curl` contra qualquer um dos três endpoints consome tokens Anthropic à custa da agência. Sem rate-limit, sem auth, sem validação de tamanho. A story #01.9 **pára a hemorragia**. A limpeza arquitectural mais profunda (`C2` bootstrap, `C1` middleware, `C5` portal token) vem em stories subsequentes.

## Acceptance Criteria

### Auth gate (primary fix)

- [ ] **AC1:** `POST /api/ai/agent`, `/api/ai/copy`, `/api/ai/automation` retornam `401 Unauthorized` quando não há sessão Supabase válida nos cookies.
- [ ] **AC2:** A verificação de sessão usa `createClient()` de `@/lib/supabase/server` (anon-role + cookies), **não** `createServiceClient()`.
- [ ] **AC3:** O check é implementado num wrapper partilhado (ex.: `lib/api/with-auth.ts` ou `lib/ai/require-session.ts`) — não três cópias espalhadas.

### Rate limiting

- [ ] **AC4:** Cada endpoint impõe rate-limit de **20 requests/minuto por IP**. A janela é sliding-window de 60 segundos.
- [ ] **AC5:** IP obtido de `x-forwarded-for` (Vercel populates), com fallback a `x-real-ip`, e rejeita com 400 se ausente.
- [ ] **AC6:** Quando o limite é excedido, resposta HTTP `429 Too Many Requests` com header `Retry-After` em segundos.
- [ ] **AC7:** Implementação suporta dois backends — Upstash Redis se `UPSTASH_REDIS_REST_URL` e `UPSTASH_REDIS_REST_TOKEN` estiverem definidos; caso contrário usa counter baseado em Supabase (tabela `ai_rate_limits`). O utilizador deve decidir qual ativar na configuração de produção (ver Technical Notes).

### Size limits e validação

- [ ] **AC8:** Zod schema por rota:
  - `/api/ai/agent`: `messages` (1-20 items, cada `content` ≤ 4000 chars); `context` (opcional, campos string ≤ 500 chars cada); `mode` (enum `'chat' | 'autonomous'`).
  - `/api/ai/copy`: `product` ≤ 200 chars; `audience` ≤ 200 chars; `objective` e `tone` enums literais do ficheiro actual; `channel` enum `'email' | 'whatsapp' | 'sms'`.
  - `/api/ai/automation`: `prompt` 1-2000 chars.
- [ ] **AC9:** Body que falha o schema → `400 Bad Request` com mensagem específica (Zod error formatted).
- [ ] **AC10:** Body maior que 16 KB antes do JSON parse → `413 Payload Too Large`. (Safety net para DoS; Vercel default é ~4.5 MB mas não queremos sequer parsear 1 MB de lixo.)

### Mock fallback — dev-only

- [ ] **AC11:** A branch "mock response" nos três ficheiros só pode executar quando **ambas** as condições são verdadeiras:
  1. `process.env.NODE_ENV === 'development'`
  2. `process.env.NEXT_PUBLIC_DEMO_MODE === 'true'`
- [ ] **AC12:** Em produção (qualquer deployment onde `NODE_ENV !== 'development'`), `ANTHROPIC_API_KEY` ausente → `500 Internal Server Error` com mensagem genérica `{"error":"AI service not configured"}` e log estruturado server-side.
- [ ] **AC13:** A detecção de placeholder via `url.includes('placeholder')` **não é usada** aqui (C1/C2 trata disso separadamente — não contaminar esta story).

### Limpeza

- [ ] **AC14:** Mensagens de erro ao cliente **não** incluem texto bruto de erros Anthropic (o route actual faz `console.error(err)` + `return ...error: String(err)...` em alguns sítios; substituir por log estruturado server-side + resposta genérica ao cliente).
- [ ] **AC15:** O `system` prompt deixa de receber valores de `context` interpolados directamente (prompt-injection surface). Valores atacante-controlados passam como `role: 'user'` separado, marcado como `CONTEXTO:` para o modelo distinguir. O system prompt original passa a ser 100 % confiável.

## Technical Notes

### Decisão de backend para rate-limit (pedir confirmação ao user antes de implementar)

**Opção A — Upstash Redis (recomendado):**
- Pros: standard Vercel + Next.js, biblioteca `@upstash/ratelimit` ready-to-use, baixa latência, tier grátis razoável.
- Contras: requer criar conta Upstash + configurar 2 env vars em Vercel.
- Custo: free tier cobre ~10k commands/day; paid tier começa em $0.2/100k.

**Opção B — Supabase postgres counter:**
- Pros: zero novas dependências externas, usa infra existente.
- Contras: cada call ao rate-limit = uma round-trip ao Postgres (adiciona latência ao path crítico), cleanup job necessário, aumenta carga na DB.
- Custo: zero marginal.

**A decidir com o user na story draft.** Código do wrapper pode suportar ambos com detecção automática via env vars.

### Exemplo do wrapper `requireSession`

```ts
// apps/hypeflow/lib/api/with-session.ts
import { NextResponse } from 'next/server'
import { createClient } from '@/lib/supabase/server'

export async function requireSession() {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) {
    return {
      response: NextResponse.json({ error: 'Unauthorized' }, { status: 401 }),
      user: null,
      supabase: null,
    } as const
  }
  return { response: null, user, supabase } as const
}
```

### Exemplo de rate-limit com Upstash

```ts
// apps/hypeflow/lib/api/rate-limit.ts
import { Ratelimit } from '@upstash/ratelimit'
import { Redis } from '@upstash/redis'

const hasUpstash = !!(process.env.UPSTASH_REDIS_REST_URL && process.env.UPSTASH_REDIS_REST_TOKEN)

const limiter = hasUpstash
  ? new Ratelimit({
      redis: Redis.fromEnv(),
      limiter: Ratelimit.slidingWindow(20, '60 s'),
      analytics: false,
      prefix: 'hypeflow-ai',
    })
  : null

export async function rateLimit(ip: string, key: string) {
  if (limiter) {
    const { success, reset } = await limiter.limit(`${ip}:${key}`)
    return { allowed: success, retryAfter: success ? 0 : Math.ceil((reset - Date.now()) / 1000) }
  }
  // Supabase fallback — implementação em Task 3 se Upstash não for usado.
  return supabaseRateLimit(ip, key)
}
```

function getClientIp(req: NextRequest): string | null {
  const xff = req.headers.get('x-forwarded-for')
  if (xff) return xff.split(',')[0].trim()
  const xri = req.headers.get('x-real-ip')
  if (xri) return xri.trim()
  return null
}

### Refactor exemplo (agent route, compact form)

```ts
export async function POST(req: NextRequest) {
  // 0. Size guard
  const cl = Number(req.headers.get('content-length') ?? '0')
  if (cl > 16_384) return NextResponse.json({ error: 'Payload too large' }, { status: 413 })

  // 1. Auth
  const { response: authRes, user } = await requireSession()
  if (authRes) return authRes

  // 2. Rate limit
  const ip = getClientIp(req)
  if (!ip) return NextResponse.json({ error: 'Missing client IP' }, { status: 400 })
  const { allowed, retryAfter } = await rateLimit(ip, 'agent')
  if (!allowed) {
    return NextResponse.json({ error: 'Rate limit exceeded' }, {
      status: 429,
      headers: { 'Retry-After': String(retryAfter ?? 60) },
    })
  }

  // 3. Parse + validate
  const rawBody = await req.json().catch(() => null)
  const parsed = agentRequestSchema.safeParse(rawBody)
  if (!parsed.success) return NextResponse.json({ error: 'Invalid request', issues: parsed.error.flatten() }, { status: 400 })

  const body = parsed.data

  // 4. Demo fallback — dev only
  const apiKey = process.env.ANTHROPIC_API_KEY
  if (!apiKey) {
    if (process.env.NODE_ENV === 'development' && process.env.NEXT_PUBLIC_DEMO_MODE === 'true') {
      return NextResponse.json({ content: getMockResponse(body.messages, body.context), demo: true })
    }
    console.error('[ai/agent] ANTHROPIC_API_KEY missing in production')
    return NextResponse.json({ error: 'AI service not configured' }, { status: 500 })
  }

  // 5. Build messages — context is user-role, NOT system
  const messages = [
    ...(body.context ? [{ role: 'user' as const, content: renderContextBlock(body.context) }] : []),
    ...body.messages,
  ]
  const system = SYSTEM_PROMPT + (body.mode === 'autonomous' ? '\n\nMODO: AUTÓNOMO...' : '')

  // 6. Anthropic call (unchanged shape)
  try {
    const r = await fetch('https://api.anthropic.com/v1/messages', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'x-api-key': apiKey, 'anthropic-version': '2023-06-01' },
      body: JSON.stringify({ model: 'claude-sonnet-4-6', max_tokens: 1024, system, messages }),
    })
    if (!r.ok) {
      const errText = await r.text()
      console.error('[ai/agent] Anthropic error', { status: r.status, body: errText, userId: user.id })
      return NextResponse.json({ error: 'AI service unavailable' }, { status: 502 })
    }
    const data = await r.json()
    return NextResponse.json({ content: data.content?.[0]?.text ?? '' })
  } catch (err) {
    console.error('[ai/agent] exception', { err, userId: user.id })
    return NextResponse.json({ error: 'Internal error' }, { status: 500 })
  }
}
```

### Concerns Registados

**CONCERN C01 (Médio) — 0 testes automatizados**
Baseline mantém-se 0 %. A regra do @qa customization é CONCERNS automático para qualquer story sem testes, a menos que seja genuinamente não-testável. Esta story *é* testável (3 endpoints HTTP, comportamentos determinísticos). O trade-off foi documentado e aprovado explicitamente pelo utilizador: story 01.10 escreve os testes como primeiros exemplos da nova infraestrutura.
- **Acção:** Story 01.10 deve incluir ≥ 4 testes por rota (201 autenticado, 401 sem sessão, 429 rate-limit, 413 tamanho) como condição de DoD.

**CONCERN C02 (Baixo) — Race condition no rate-limiter**
`count-then-insert` não é atómico. Dois requests concorrentes no boundary podem ambos passar. Documentado nos comentários do código (`rate-limit.ts:20-24`). Para o modelo de ameaça desta story (bloquear chamadas anónimas em loop), um ou dois requests extras no boundary não representam risco financeiro. A auth gate mantém-se.
- **Acção:** Considerar `SELECT ... FOR UPDATE` ou função Postgres em story futura se o tráfego crescer.

**CONCERN C03 (Baixo) — Tabela `ai_rate_limits` sem cleanup**
Rows acumulam indefinidamente. A migração documenta que é trabalho futuro. Em produção com 20 req/min, uma agência activa pode gerar ~28.800 rows/dia. Não é crash imediato mas é tech debt operacional.
- **Acção:** Adicionar Edge Function de cleanup (DELETE WHERE ts < now() - interval '10 minutes') em story de manutenção. Pode ser 01.10 ou story separada.

**CONCERN C04 (Informativo) — `content-length` spoofável**
O guard de 16 KB usa o header `Content-Length`, que pode ser ausente (chunked encoding) ou manipulado. Quando ausente, `Number('0') = 0` → guard passa, e o body real lê-se sem limite. Em Vercel, o runtime tem os seus próprios limites (~4.5 MB por route handler), mas isso é muito maior que 16 KB. Vercel Edge Middleware ou um `AbortController` timeout no `req.json()` seriam mais robustos.
- **Acção:** Informativo — monitorizar se há casos de bypass em produção. Não bloqueia esta story.

**CONCERN C05 (Baixo) — Task 11 (curl) não executada**
A verificação manual dos 401 não foi feita porque requer servidor local a correr. Typecheck passou limpo; comportamento do auth gate é deduzível do código. No entanto, o DoD explícito exige verificação manual.
- **Acção:** O utilizador deve executar antes do push: `curl -i -X POST http://localhost:3000/api/ai/agent -H 'Content-Type: application/json' -d '{"messages":[{"role":"user","content":"hi"}]}'` e confirmar `HTTP/1.1 401`.
